Auth, Accounts & Database Safety — Twin Scene
Operational context for coding agents in this repo. (Fuller design rationale lives in the team's Claude Project; this file is what you need day-to-day.) For the fuller auth/profile design + status narrative, see docs/architecture.md.
Auth & accounts model
* ONE users table is the single identity. Authorization is a SEPARATE layer:
    * users.is_admin → may edit ANY band. Granted/revoked by another admin from /admin/users (setUserAdmin in lib/auth.ts, PATCH /api/admin/users/[id]) — no longer a by-hand DB flip. Two guards: you can't demote yourself (the route would leave you with no way back in), and the last remaining admin can't be demoted (a SELECT … FOR UPDATE over the admin rows, so concurrent demotions can't race the site down to zero admins).
    * band_editors(user_id, band_id, role) → which non-admin users may edit which bands (many-to-many). role is 'editor' or 'owner'; an 'owner' (earned by redeeming an admin-issued ownership code) may ALSO approve that band's member claims, which is how approval delegates out of the admin queue.
* Permission check, ALWAYS server-side: canEditBand(user, bandId) = user.is_admin OR row in band_editors(user_id, band_id). Never gate on hidden UI — a missing button is not a permission check.
* TWO login methods, both landing on the SAME session (createSession issues an opaque sessions token in an HTTP-only, Secure, SameSite=Lax cookie; getCurrentUser() reads it — this layer is method-agnostic and unchanged):
    * Magic link (passwordless, the original + default): login_tokens (store a HASH of the token, single-use, ~15 min expiry). Logging in creates the user row — that IS signup for this path. Clicking the link also stamps email_verified_at (ownership proven), so every magic-link user is verified.
    * Email + password (migration 0038): users.password_hash is argon2id (via @node-rs/argon2, lib/password.ts — never plaintext). Password LOGIN is gated on users.email_verified_at, so a password signup must confirm its email first; magic-link users are implicitly verified and unaffected. password_hash is NEVER exposed: lib/users.ts scrubUser strips it out of every User the app hands around and substitutes a derived has_password boolean. The two are one identity — a user can use either method on the same account.
* login_tokens carries a `type` ('login' | 'verify' | 'reset') so the one hashed/single-use/~15-min mechanism backs all three emailed links. Consumption is type-scoped (consumeToken) — a reset link can't be redeemed as a login, etc. Existing rows default to 'login'.
* Password flows (all server-side, generic errors, no user-enumeration):
    * Signup (POST /api/auth/signup): creates an UNVERIFIED user with a password + emails a type='verify' token. Always returns a generic success; never overwrites an existing account's password (takeover guard — existing accounts adopt a password via reset instead). Clicking the verify link (/api/auth/verify) stamps email_verified_at and starts a session.
    * Login (POST /api/auth/password-login): argon2id-verify against password_hash, then createSession like the magic-link path. Generic "invalid email or password" for unknown email / no password set / mismatch (timing equalized against a dummy hash so those cases are indistinguishable); "verify your email" is revealed ONLY after a correct password (can't enumerate). Failure-based lockout per (email, ip) via login_attempts (lib/loginAttempts.ts): N failures in a window → 429; a success clears the streak.
    * Forgot / reset (POST /api/auth/forgot → /reset page → POST /api/auth/reset): emails a type='reset' token; redeeming it sets a new password_hash AND marks the email verified. This is ALSO how a password-less magic-link user adopts a first password.
    * Set/change while logged in (POST /api/auth/password, from /profile/edit): session proves ownership; changing an existing password requires the current one, setting a first one doesn't.
* band_claims = a user requests a band; an admin approves, which inserts the band_editors row AND marks the claim decided in ONE transaction.
* band_member_claims = "I'm <musician> in <band>" — band-scoped, approved by that band's owner (admin fallback when the band has no owner). Check canApproveMemberClaim server-side. Supersedes the dropped musician_claims table, which granted rights on every band a musician appeared in.
* Ownership codes (band_ownership_codes) follow the login_tokens rule: store only a HASH, single-use, expiring. The plaintext exists only in the admin's one-time generate response.
* Validate any login next/redirect param as same-origin relative only (open-redirect guard).
* Key files: lib/auth.ts (sessions, getCurrentUser, canEditBand, isAdmin, findOrCreateUserByEmail, and the password/token helpers: createEmailToken, consumeToken, createPasswordUser, getUserAuthByEmail, markEmailVerified, setPasswordByEmail, setPasswordForUser), lib/password.ts (argon2id hash/verify/validate + DUMMY_PASSWORD_HASH), lib/loginAttempts.ts (failure-based lockout), lib/authRateLimit.ts (email-send throttle + clientIp), lib/users.ts scrubUser (strips password_hash), lib/email.ts, app/api/auth/* (login, callback, signup, verify, password-login, forgot, reset, password, logout), lib/bandEditors.ts, lib/bandClaims.ts, lib/bandFollows.ts (the heart — lib/savedBands.ts is GONE, merged in migration 0028), lib/bandOwnership.ts, lib/bandMemberClaims.ts, lib/musicians.ts, lib/notifications.ts, lib/feed.ts, admin UI under app/admin/* — the whole dashboard now gates on is_admin (the shared AdminNav in app/admin/layout.tsx + lib/fetchUsers.ts back the users view). SCRAPE_SECRET is no longer a page gate anywhere; it survives ONLY as the machine token for the scrape/show APIs, the daily cron, and scrape:local. The is_admin-gated scraper pages still read SCRAPE_SECRET server-side and hand it to their client panels so those API calls authenticate.
Email
* Sends via Resend (HTTP fetch, no SDK). lib/email.ts logs the link to the console in dev when RESEND_API_KEY is unset, and throws in prod if unset.
* EMAIL_FROM = "Twin Scene <login@thebirdhaus.org>" — Resend free tier allows one verified domain, so Twin Scene borrows Birdhaus's verified domain (display name still reads "Twin Scene"). RESEND_API_KEY + EMAIL_FROM live in Vercel Production ONLY, not in local .env.local — so local logins print the magic link to the dev console.
DATABASE — READ BEFORE ANY MIGRATION OR WRITE
Neon Postgres. This repo shares its DB with Crawlspace (canonical bands, canonical venues, shared shows, all auth/user tables). Birdhaus has its OWN separate DB. Column types: shows.id is uuid (gen_random_uuid()); bands.id / venues.id / users.id are bigint.
Dev/prod isolation — DO NOT UNDO:
* .env.local points at the Neon DEV branch. .envrc uses dotenv_if_exists .env.local so direnv watches the file and the shell can't go stale. Both files are gitignored.
* The shell's DATABASE_URL (exported by direnv) = DEV. scripts/migrate.mjs uses Node's process.loadEnvFile('.env.local'), which does NOT override an already-set env var — so it targets whatever the shell holds = DEV by default. This is intentional.
* scripts/whichdb.mjs prints which DB the repo is pointed at (host + current_database, and warns if the shell disagrees with .env.local). RUN IT before any write.
Rules:
* NEVER run test writes against prod. Test against the dev branch, or a seeded throwaway row with a guard that hard-fails if the target isn't the throwaway. Delete test rows after and verify zero leftovers.
* To target PROD deliberately (e.g. run a migration on prod), run `npm run migrate:prod` (scripts/migrate-prod.mjs). It reads the prod connection string from `.env.prod.local` (gitignored, one-time setup — see the script's header comment for where to get the value; `vercel env pull` won't work, it redacts sensitive vars), prints the connected host + database, and requires typing "yes" before it applies anything (`--yes` skips the prompt). Falling back to the manual one-off prefix — `DATABASE_URL='<prod-url>' node scripts/migrate.mjs` — still works if `.env.prod.local` isn't set up. Either way: confirm the printed host is PROD, not dev, before applying. Never commit the prod URL or paste it into a chat/session transcript.
* Migrations are additive and sequential (scripts/migrations/NNNN_*.sql). Apply new migrations to PROD BEFORE deploying code that reads the new tables.
Deploy to prod
1. Confirm you're on dev via whichdb.mjs. Commit only the intended files (leave unrelated untracked work alone).
2. Apply new migration(s) to prod: `npm run migrate:prod` (see above). Verify the host is prod; sanity-check it was additive (e.g. existing row counts unchanged).
3. Push to main — Vercel builds production from main.
4. Verify on the live site (twinscene.org).

* Unified messaging, slice 1 (migration 0053) — SHIPPED to prod. One inbox per user (/profile/messages) aggregating conversations addressed to any identity they hold: bands they edit (band_editors) + their linked musician (musicians.user_id), each thread badged with the addressed identity. New tables: conversations (polymorphic recipient_type/recipient_id, 'band'|'musician' — 'user' reserved for a later user-to-user slice, deliberately NOT exposed yet), conversation_participants (the initiating human), messages (sender_as_type/id = the identity a reply was sent as; sender_user_id always the real human, so per-editor attribution is addable with no schema change), conversation_reads (per-person read tracking — one editor reading doesn't mark it read for the others). Key file lib/messaging.ts. Auth mirrors the app: canEditBand for bands, musicians.user_id for musicians; inbox aggregation uses actual band_editors rows (NOT canEditBand) so admins don't get every band's DMs, but thread viewing still admits admins; non-participants get 404, not 403. KNOWN GAP: no blocking / reporting / rate-limiting on sending — revisit anti-abuse before any push that drives strangers to message at volume.
* Unified messaging, slice 2 (migration 0054) — SHIPPED to prod. In-app notifications for new messages, reusing the notify-on-write + join-on-read system (0026) with a 4th type 'new_message'. Migration is additive: notifications.conversation_id (nullable uuid, on delete cascade) + a (user, conversation) unread coalesce index (same shape as band_update/show_changed). lib/notifications.notifyNewMessage fans out inside sendMessage's transaction to everyone who can see the thread EXCEPT the sender (band editors / linked musician + the initiator); a burst of replies coalesces to one unread ping (snippet + timestamp track the latest). listNotifications resolves POV: the initiator sees "New message from {band/musician}", the recipient side sees "{initiator} messaged {band/musician}". Delivery is in-app only — the email-digest seam 0026 left open is still unused (a candidate next slice, alongside the messaging anti-abuse gap above).
